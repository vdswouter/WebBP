Colors

Primary colors

Always provide different shades of the primary color, Google Material Design is a good starting point for this. Also note that black should never be just black (#000), adding some warm/cold colors to the mix will make it more contemporary.

Text colors

Use different shades for text to express the hierarchy of information. Text, icons, borders and dividers will benefit when their color is defined with alpha values, as a result they will work on any background. Text colors are derived from primary black/white colors.

Typography

General principle

Using system fonts over Google fonts will benefit the loading time of the application/site. Note that the use of custom webfonts has a negative impact on loading if not delivered through a Content Delivery Network, especially on mobile and low-end devices. Less is more, so keep the plethora of font families and weights to a reasonable minimum.

Suggested Google fonts: Source Sans Pro, Lato, PT Sans, Open Sans

Font stacks

Sans-serif: Helvetica Neue, Helvetica, Arial, sans-serif
Serif: Lora, serif

Images

General principle

Imagery should be used to enhance user experience and express personal relevance and/or information. Try to avoid the use of stockphotos at all times. Images are set to be fluid by default, unless width and/or height are specified.

Contextual images

When used in the context of a page/article, images should be wrapped in a <figure>, captions should always be provided underneath.

Overlays

For use in headers, heroes and/or mastheads. Image overlays should have an opacity between 20-60% depending on context. Use font-color/border-color values for this.

Buttons

General principle

Choosing a button type depends on the importance of the interaction. Primary buttons should be used for key-interactions and navigation, secondary buttons for in-context interaction (eg. show/hide, load more) and flat buttons for smaller, grouped interactions and/or in modal windows.

Implementation

Use <a> for navigation, <button> for everything else. Submit/reset buttons are the only exceptions, using <input[type="submit"]> to trigger the default browser behaviour associated with form input.

Button sizes

Recommended button height should be 5 × 1 base-line grid-unit
Base: 40px (5 × 8px)
Desktop: 45px (5 × 9px)
Large desktop: 50px (5 × 10px)

In-context actions

Buttons for small in-line interaction (eg. close, add, remove, ...). In most cases these will be based on icons provided in the icon-font. If possible however, use HTML-enitities as this saves some icon-characters (bytes).

Feedback

General principle

Always provide feedback on user interaction. This can be immediate (eg. inline validation) or after (eg. submit a form). Feedback messages are informative and provide an easy escape or alternative/following action. The tone of voice should be human and add personality to the experience/brand.

Implementation

Button interactions to remove the alert are provided by default. To enable the actual functionality you will need to include /js/remove-dom-elements.js in the page. Degrades gracefully when javascript is not enabled in the client by hiding the buttons with CSS.
Default: <div class="alert">

Hi there!

This is a feedback message.

Error: <div class="alert error">

Oops!

Something went wrong, please try again.

Warning: <div class="alert warning">

Warning!

Perhaps you meant something else.

Success alert: <div class="alert success">

Great!

Now how about adding some more.

Information alert: <div class="alert information">

Information!

Some features might not work.

Tooltips

Use tooltips to display inline information or validation feedback. Just like default alerts, different styling can be applied (error, warning, success and information). Although tooltips will work with any typographic style, the use of <small> is adviced.

General markup: <small class="tooltip-xx state">
